# Client Probe Tool

`xaccel-client-probe` turns the manual validation flow into one command. It is
not the final game client. It is a deployment and integration diagnostic tool
for node operators and backend developers.

Version `0.37.0` can run directly from a per-session `accel_ticket` returned by
the business backend, send the ticket `route_policy` during probe, and relay a
`session.data` packet with explicit target metadata. Version `0.33.0` can send
the optional `XACCEL_CLIENT_API_TOKEN` required by a control plane that protects
the legacy client `connect-intent` endpoint.

## What It Checks

- Control API is reachable.
- MySQL route and node metadata can issue a connect-intent.
- The selected node receives UDP traffic.
- The node accepts the `xat.v1` token generated by control-api.
- The node validates `route_policy_hash` when the ticket contains a dynamic
  route policy.
- The node creates a short-lived UDP session.
- The node forwards `session.data` to the signed route target or to an explicit
  target allowed by `route_policy.targets[]`.

## Usage

Run it on any machine that can reach both the control API and the node UDP port:

```bash
xaccel-client-probe \
  --control-url http://127.0.0.1:18080 \
  --client-api-token "${XACCEL_CLIENT_API_TOKEN}" \
  --user-id 1001 \
  --device-id pc-001 \
  --game-id 8888 \
  --platform pc \
  --client-isp telecom \
  --client-ip 127.0.0.1 \
  --bandwidth-quality fast \
  --payload hello
```

Per-session ticket mode:

```bash
xaccel-client-probe \
  --accel-ticket-file ./accel-ticket.json \
  --target-host 198.51.100.20 \
  --target-port 27015 \
  --target-id naraka-gameplay-observed \
  --payload hello
```

When `route_policy.targets[]` includes a concrete UDP host, the probe can derive
the first target automatically. Pass `--target-host` and `--target-port` for
`host_type=any`, Steam/SDR-style port checks, or a specific client-observed IP.

For the current two-server test, run it on the control server `s54112`:

```bash
xaccel-client-probe \
  --control-url http://127.0.0.1:18080 \
  --client-api-token "${XACCEL_CLIENT_API_TOKEN}" \
  --user-id 1001 \
  --device-id pc-001 \
  --game-id 8888 \
  --client-isp telecom \
  --client-ip 127.0.0.1 \
  --bandwidth-quality fast
```

The output is JSON so scripts and future dashboards can consume it directly.

## Useful Options

- `--client-api-token TOKEN`: sends `Authorization: Bearer TOKEN` to
  `/api/client/v1/connect-intent`. Only needed when the control panel is
  installed with `--client-api-token`.
- `--skip-session-data`: only checks connect-intent and UDP probe.
- `--target-host HOST` and `--target-port PORT`: sends explicit
  `session.data.target` metadata for dynamic route-policy checks.
- `--target-id ID`: optional `route_policy.targets[].target_id` hint.
- `--target-protocol udp`: target protocol. UDP is the only relay mode
  currently implemented.
- `--payload TEXT`: sends `TEXT` as the session payload. Default is `hello`.
- `--response-timeout-ms N`: upstream UDP wait time. Default is `500`.
- `--candidate-index N`: selects a non-first candidate if the API returns many.
- `--compact`: prints a single-line JSON result.

## Reading Results

Key fields:

- `status`: `ok` means the whole requested flow completed.
- `node.address`: selected node endpoint.
- `probe.credential_valid`: node accepted the control-api token.
- `probe.session_id`: short-lived node session id.
- `session_data.status`: `forwarded`, `echo`, or `upstream_timeout`.
- `session_data.relay.timed_out`: whether the route target responded in time.

If the node returns `invalid_token`, the control-api database `node_secret` does
not match the node identity secret. Re-read `/var/lib/xaccel-node/identity.json`
on the node and update `accel_nodes.node_secret`.
